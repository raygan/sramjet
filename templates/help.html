Setting Up a New Device ⌃

Settings › Saving

Sort Saves and States into folders by core name — leave this enabled (it's the default). SRAMjet uses the folder structure to group files by game in the dashboard. All devices must use the same setting, or the same game's files will appear under different names on different devices.

Sort Saves by Content Directory organizes saves by ROM folder name instead of core name, which will conflict with the core-name grouping used by default on most devices. Write Saves to Content Dir moves saves out of the saves/ folder entirely and stores them alongside your ROMs. Both must be disabled (the default).

Use a unique device name for each device. SRAMjet registers new devices automatically on first sync. The name appears in the dashboard and is used to track which device last touched each file.

Security: SRAMjet has no authentication. Do not expose it directly to the internet. Run it on a private network, or use a VPN like Tailscale or a zero-trust tunnel like Cloudflare Access.
{# ── Resolving Sync Issues ─────────────────────────────────────────────────── #}
Resolving Sync Issues ⌃

My device has an updated save but it isn't uploading

RetroArch performs its own conflict check before uploading. If it detects that both the local file and the server file have changed since the last sync, it will refuse to upload and report "cloud sync completed with conflicts" — even if the server would have accepted the file just fine.

Fix: Go to the Devices page and click Trust Next Sync for that device. On the next sync, RetroArch will treat the server as a clean slate and re-upload all local files unconditionally. The flag clears automatically once the sync completes.

The server has a newer save but my device isn't downloading it

The same client-side conflict check can block downloads too. If RetroArch sees that both local and server have changed since it last synced, it may skip the download entirely.

  • If you want the local version to win — use Trust Next Sync. The device will re-upload its local files and they'll become the new canonical versions on the server.
  • If you want the server version to win — go to Files, find the save, and click Revert on the version you want to restore. Then sync the device normally. If it's still stuck, use Trust Next Sync after the revert.

How SRAMjet handles conflicts

Standard WebDAV servers have no awareness of sync conflicts — they just store whatever is uploaded. RetroArch's conflict detection is entirely client-side, with no way for the server to communicate resolution hints back.

SRAMjet resolves this by tracking full version history and using last-write-wins: whichever device uploads a file last becomes the canonical version, and the displaced version is kept in history so you can revert if needed. The server never gets into a stuck state.

The Trust Next Sync feature works by serving an empty manifest to the affected device. RetroArch interprets an empty manifest as a server that has been reset and unconditionally re-uploads all local files, bypassing its client-side conflict logic. Under the hood, this is the same path RetroArch uses on a brand-new device's first sync.

{# ── Syncing ROMs ──────────────────────────────────────────────────────────── #}
Syncing ROMs ⌃

RetroArch's cloud sync is not designed for ROMs. It works best when the total file size is small — large ROM libraries will make every sync slow and can time out.

That said, if you set up new devices often or want a small shared collection available everywhere, a practical workaround is to create a subfolder inside RetroArch's system/ directory:

retroarch/system/roms/NES/
retroarch/system/roms/GB/

Enable Sync System Files on any device where you want access to these ROMs. SRAMjet will store them and serve them to other devices that have system file syncing on. Keep this collection small — a few dozen files at most.
