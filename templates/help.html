Setting Up a New Device ⌃

Adding a device to sync doesn't require any server-side setup and most RetroArch settings can stay at their defaults. Here are a few settings to enter, and some best practices.

Settings › Saving

Setting Value Status
Sort Saves into Folders by Core Name On Default
Sort States into Folders by Core Name On Default
Sort Saves by Content Directory Off Default
Sort States by Content Directory Off Default
Write Saves to Content Directory Off Default
Don't Overwrite Save File on Loading State On Change
Auto Save State On Recommended
Auto Load State On Recommended
Save State Thumbnails On Recommended

All devices must use the same folder sorting settings. Mixing them causes the same game's files to appear under different names on different devices.

Use a unique device name for each device. SRAMjet registers new devices automatically on first sync. The name appears in the dashboard and is used to track which device last touched each file.

Security: HTTP Basic auth is available for the dashboard and WebDAV endpoints — set AUTH_UI_USERNAME / AUTH_UI_PASSWORD and AUTH_WEBDAV_USERNAME / AUTH_WEBDAV_PASSWORD in your environment to enable them independently. Even with auth enabled, Basic auth over plain HTTP sends credentials in the clear — use HTTPS, a VPN like Tailscale, or a zero-trust tunnel like Cloudflare Access if exposing SRAMjet outside your local network.
{# ── Resolving Sync Issues ─────────────────────────────────────────────────── #}
Resolving Sync Issues ⌃

My device has an updated save but it isn't uploading

RetroArch performs its own conflict check before uploading. If it detects that both the local file and the server file have changed since the last sync, it will refuse to upload and report "cloud sync completed with conflicts" — even if the server would have accepted the file just fine.

Fix: Go to the Devices page and click Re-upload All Files for that device. On the next sync, RetroArch will treat the server as a clean slate and re-upload all local files unconditionally. The flag clears automatically once the sync completes.

The server has a newer save but my device isn't downloading it

The same client-side conflict check can block downloads too. If RetroArch sees that both local and server have changed since it last synced, it may skip the download entirely.

  • If you want the local version to win — use Re-upload All Files. The device will re-upload its local files and they'll become the new canonical versions on the server.
  • If you want the server version to win — go to Files, find the save, and click Revert on the version you want to restore. Then sync the device normally. If it's still stuck, use Re-upload All Files after the revert.

How SRAMjet handles conflicts

Standard WebDAV servers have no awareness of sync conflicts — they just store whatever is uploaded. RetroArch's conflict detection is entirely client-side, with no way for the server to communicate resolution hints back.

SRAMjet resolves this by tracking full version history and using last-write-wins: whichever device uploads a file last becomes the canonical version, and the displaced version is kept in history so you can revert if needed. The server never gets into a stuck state.

The Re-upload All Files feature works by serving an empty manifest to the affected device. RetroArch interprets an empty manifest as a server that has been reset and unconditionally re-uploads all local files, bypassing its client-side conflict logic. Under the hood, this is the same path RetroArch uses on a brand-new device's first sync.

{# ── MiSTer FPGA Sync ──────────────────────────────────────────────────────── #}
Syncing with a MiSTer FPGA ⌃

SRAMjet can sync game saves with a MiSTer FPGA. A small script runs on the MiSTer (no dependencies — MiSTer ships Python 3) and talks to this server. SRAMjet converts save formats between MiSTer cores and emulator cores automatically, so you can play a game in RetroArch, then continue on the MiSTer, and back. The MiSTer appears on the Devices page like any other device.

Install

  1. Download a pre-configured copy of the sync script:
    The device name identifies this MiSTer in the dashboard.
  2. Copy it to /media/fat/Scripts/sramjet_sync.py on the MiSTer (over SMB or SCP).
  3. Create /media/fat/Scripts/sramjet_sync.sh so it shows up in the MiSTer Scripts menu: 
    #!/bin/bash
python3 /media/fat/Scripts/sramjet_sync.py
  4. Run sramjet_sync from the MiSTer's Scripts menu after playing. Add --dry-run on the first run to preview what would happen.

If WebDAV auth is enabled on this server, edit AUTH_USER / AUTH_PASS at the top of the downloaded script.

Each system syncs with one RetroArch core folder. To use different cores, set the MISTER_CORES environment variable on the server, e.g. MISTER_CORES=snes=bsnes,gba=VBA-M.

Notes and limitations

  • Saves only — save states are emulator-specific and can't work on a MiSTer.
  • N64, Saturn, and Sega CD are not yet supported (complex multi-part save formats).
  • Game Boy saves are written to all three MiSTer folders (GAMEBOY, GBC, SGB) since the mapping can't tell which core a game uses. Extra copies are harmless.
  • On the very first sync, if a game exists on both sides with different progress, the most recently modified side wins. Make sure the MiSTer's clock is set (it syncs via NTP when networked).
  • Conflicts after that are resolved last-write-wins, same as RetroArch devices — history is kept, so anything can be reverted from the Files page.
{# ── Syncing ROMs ──────────────────────────────────────────────────────────── #}
Syncing ROMs ⌃

RetroArch's cloud sync is not designed for ROMs. It works best when the total file size is small — large ROM libraries will make every sync slow and can time out.

That said, if you set up new devices often or want a small shared collection available everywhere, a practical workaround is to create a subfolder inside RetroArch's system/ directory:

retroarch/system/roms/NES/
retroarch/system/roms/GB/

Enable Sync System Files on any device where you want access to these ROMs. SRAMjet will store them and serve them to other devices that have system file syncing on. Keep this collection small — a few dozen files at most.
